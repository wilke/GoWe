GoWe compiles Common Workflow Language into a DAG, drives it through a tick-based scheduler, and dispatches concrete work units across five interchangeable backends — local, Docker, Apptainer, remote pull-workers, and BV-BRC. Each section below takes one layer: what it does, the entities it moves, and the exact states, phases, and endpoints involved.

Companion to SPECIFICATION.md (the normative contract) and docs/adr/ (the decisions behind this design). Scope: the Go implementation as it stands today. Every state, phase, and route below was verified against the source — state constants in pkg/model, the tick loop in internal/scheduler/loop.go, and routes in internal/server.

03

The scheduler tick

~2s loop · six phases (+ sub-phases)
  1. 1Advance waiting — WAITING → READY once every dependency is met.

    1.5 · pre-stage workspace inputs for pending submissions (server mode).

  2. 2Dispatch — resolve inputs, create Tasks, submit to executors.

    2.5 · re-submit RETRYING tasks.

  3. 3Poll — query QUEUED/RUNNING async tasks for status.

    3.5 · detect stuck queued worker tasks by progress.

  1. 4Advance steps — move DISPATCHED/RUNNING instances forward when all their tasks are terminal.
  2. 5Finalize — settle submissions once every step instance is terminal.

    5.5 · post-stage outputs to the workspace (server mode).

  3. 6Retries — send newly-FAILED tasks to RETRYING while attempts remain.

Every ~5s the worker requests a task matched on runtime, group, and dataset affinity.
GET /workers/{id}/work

internal/toolexec

Periodic heartbeat keeps the worker registered; the scheduler flags stalled claims.

Credentials & auth

four planes · one trust boundary

GoWe keeps four credential planes separate, and one rule holds them together: execution secrets never travel back to the server. Identity is delegated to external providers — GoWe stores no passwords.

User / API token client → server
A BV-BRC or MG-RAST pipe-delimited token on Authorization or X-MG-RAST-Token. Parsed for username + expiry; the account is auto-created on first use.
internal/server/auth.go
Worker key worker → server
A shared X-Worker-Key mapped to allowed worker groups. Optional — open access if unset — and logged only as a SHA-256 hash, never in the clear.
internal/server/worker_auth.go
Worker secrets worker-local
--secret NAME=value / --secret-file injected into every container's env. Held in worker memory only — never sent to the server; values never logged.
internal/worker → internal/toolexec
Delegated BV-BRC token server → container
The submitter's own token, injected as BVBRC_TOKEN when gowe:Execution.inject_bvbrc_token is set — so BV-BRC jobs run under the user's identity.
scheduler/loop.go → worker.go
◧Client
gowe CLI · browser
Holds the user's provider token; sends it per request.

▤Worker → container

◀  Return path carries results only — injected secrets and container env never flow back to the server.

Current-state caveats. The design is sound, but a few things are worth knowing before production: the submitter's BV-BRC token is persisted plaintext in submissions.user_token (SQLite); the worker key is a shared secret with no per-worker rotation; and server TLS is not enforced by default (Secure:false, a standing TODO) — front the server with a TLS terminator. Anonymous access is opt-in and should always be scoped with --anonymous-executors.

08

Routing hints — reference

gowe:* extensions · portable by design

CWL has two extension surfaces: requirements (the engine MUST support them or reject the document) and hints (SHOULD support, MAY ignore). GoWe puts its placement metadata in namespaced hints — so cwltool, Toil, and others skip unknown gowe:* keys instead of failing, and the workflow stays one portable artifact.

$namespaces:
  gowe: https://github.com/wilke/GoWe#

hints:
  gowe:Execution:
    executor: worker        # local · worker · bvbrc
    worker_group: esmfold
    bvbrc_app_id: GenomeAssembly2
    docker_image: override.sif
    gpu: true
    inject_bvbrc_token: true

  gowe:ResourceData:
    datasets:
      - id: boltz
        path: /local_databases/boltz
        mode: prestage    # require | cache=prefer
Placement rule
gowe:* GoWe extensions go in hints — advisory and portable. They steer where and how a step runs.
CWL std Container / resource needs use standard DockerRequirement and ResourceRequirement, honored as CWL intends.

Caveat: because hints are advisory, a workflow that depends on GoWe routing runs differently (though validly) on another engine.

gowe:Execution — steers placement · recognized only in hints
FieldTypePurpose

worker_groupstringTarget a worker pool (e.g. esmfold); overrides the submission-level --group.
bvbrc_app_idstringBV-BRC application id; implies executor: bvbrc.
docker_imagestringContainer image override — wins over DockerRequirement; feeds Docker and Apptainer.
gpuboolRequest GPU passthrough on the executing worker.
inject_bvbrc_tokenboolInject the submitter's token as BVBRC_TOKEN — job runs under the user's identity.
gowe:ResourceData — reference-data affinity · hints or requirements
FieldReqPurpose
idyesDataset identifier — must match a dataset a worker advertises.
pathnoExpected mount path inside the container.
sizenoApproximate size, informational only (e.g. 2TB).
modenoprestage = require a worker with it; cache = prefer (default).
sourcenoReserved for future on-demand caching (shock://, s3://).
Standard CWL requirements — honored, not GoWe-specific
DockerRequirementContainer image (dockerPull); GoWe extends .sif/Apptainer resolution.
ResourceRequirementCPU / RAM / disk requests.
InitialWorkDirRequirementStage files & dirs into the working directory (IWDR).

On docker_image: CWL's own DockerRequirement.dockerPull is honored when present — this hint doesn't replace it. It's an override that takes precedence, letting one step swap images at deploy time. Despite the name, the value drives both runtimes: a .sif suffix resolves to a local Apptainer image, anything else is pulled as docker://.

Inputs & storage

job inputs · six schemes · reference data

Two concerns share this layer: how a run's inputs arrive, and how File / Directory data moves in and out. A file's location scheme selects the backend — staging (pkg/staging) is pluggable and dispatches by URI scheme.

supported defined · implemented, not CI-verified planned
SchemeBackendBehaviorStatus
local · nonefile.go In place, no copy — the cwl-runner default.supported
file://shared.go Copy to a shared filesystem path — distributed workers.supported
http(s)://http stager HTTP PUT/POST upload; custom headers, auth, retry.defined
shock://shock.go Shock data service — shock://host/node/{id}.defined
ws://workspace.go BV-BRC Workspace — the default for BV-BRC runs.defined
s3://s3.go S3-compatible object storage.planned

Why ws:// is "defined", not "supported". The BV-BRC path is built end to end — submit, poll, output mapping, and server-side pre/post-staging — but it can't run in the conformance suite (it needs a live BV-BRC service and a real token), and it carries edges only visible against real infrastructure: wildcard-glob outputs, large-file streaming, and recursive directory download. Full round-trip in docs/BVBRC-Workspace-Deep-Dive.md.

Job inputs

Supplied at submit time as a YAML/JSON job document — not a configured backend. Relative File paths resolve against the job file's directory.

# CLI
gowe submit workflow.cwl --inputs job.yaml

# API
POST /api/v1/submissions  { inputs: … }
Reference data

Large read-only datasets (databases, model weights) are not File inputs. Declared with gowe:ResourceData and matched to workers by affinity:

prestage = require cache = prefer

Workers advertise datasets via --pre-stage-dir or --dataset id=path.

Server-side staging. The server can pre/post-stage ws:// itself (--workspace-staging server, --workspace-url) and proxy uploads to a backend (--upload-backend shock|s3|local); otherwise outputs are staged by the workers.
